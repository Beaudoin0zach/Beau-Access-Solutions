# Benefits Navigator — Keycloak OIDC integration (scope)

Anchors: [keycloak-prod-standup-checklist.md](keycloak-prod-standup-checklist.md) ·

Benefits Navigator is the last BAS app **not** authenticating against the platform Keycloak
(`bas` realm on `id.beauaccesssolutions.com`). Unlike Access Atlas / Disability Wiki (which
had ready OIDC seams and were flipped by config), BN has **no OIDC RP at all** — it uses
`django-allauth` (65.x) + `allauth-2fa` for local email/password login. Bringing it onto the
IdP is genuine integration work, not a domain flip.

## Recommended approach: allauth's `openid_connect` provider

**Not** `mozilla-django-oidc` (which CIT/KindredAccess use — they aren't allauth apps). BN
*is* an allauth app, so the native fit is allauth's built-in OpenID Connect provider:

- `allauth.socialaccount` is already installed — adding `openid_connect` is config, **no new
  dependency**.
- It plugs into allauth's existing login/signup/**2FA**/email-verification flows. Bolting on
  `mozilla-django-oidc` would create a parallel auth path that bypasses `allauth-2fa`
  (`ACCOUNT_ADAPTER = allauth_2fa.adapter.OTPAdapter`) — an MFA-bypass risk.

Same Keycloak `bas` realm and a **confidential** client; only the RP library differs, dictated
by BN's existing stack.

## Three decisions required BEFORE coding

1. **Where does MFA live?** BN enforces its own 2FA (`allauth-2fa`, `VSO_MFA_REQUIRED` for VSO
   staff). For SSO users, MFA should happen **at Keycloak** (centralized step-up) or users get
   double-prompted. Decide: defer MFA to Keycloak for SSO logins (read `acr`/`amr` from the
   token) vs. keep BN's local 2FA layered on top. Affects the VSO-staff gating.
2. **Existing-user linking (ADR-004).** BN has local email/password accounts. When an SSO user
   returns with the same email, allauth can auto-link — but **only safely on a verified email**,
   or it's an account-takeover vector. Decide the connect-by-verified-email policy and whether
   there's a dual-login migration window or a hard cutover.
3. **SSO primary or optional?** Keep the local login form during migration (add a "Sign in with
   Beau Access Solutions" button) or force SSO as the only path.

## Realm client (mostly done)

`benefits-navigator-web` (confidential + PKCE, pairwise `sub`, audience) is already scaffolded:
- **Stubbed** in [`identity/dev/realm/bootstrap.sh`](../../identity/dev/realm/bootstrap.sh) —
  inert until `BN_REDIRECT_WEB` is set (skips otherwise).
- `BN_PAIRWISE_SALT` + `BN_CLIENT_SECRET` are generated by
  [`identity/prod/gen-secrets.sh`](../../identity/prod/gen-secrets.sh).

To create it on the live realm, run the stub with:
- `BN_REDIRECT_WEB='https://vabenefitsnavigator.org/accounts/oidc/keycloak/login/callback/'`
  (allauth's callback path for provider_id `keycloak` — **confirm the canonical prod domain**),
- `BN_CLIENT_SECRET` from `secrets.env`.

(A full `bootstrap.sh` re-run is NOT idempotent against the live realm — use a targeted
per-client script like `create-disability-wiki-client.sh` / `enable-kindredaccess-oidc.sh`.)

## App code (`Beaudoin0zach/benefits_navigator`)

Keep it **inert by default** (mirror KindredAccess's `oidc_config.py` guard pattern):

- `INSTALLED_APPS += ["allauth.socialaccount.providers.openid_connect"]` (only when configured).
- `SOCIALACCOUNT_PROVIDERS["openid_connect"]` → one `keycloak` app:
  - `settings.server_url = f"{KEYCLOAK_ISSUER}/.well-known/openid-configuration"`,
  - `client_id` / `secret` from env,
  - `"OAUTH_PKCE_ENABLED": True`.
- Guard the whole block on `KEYCLOAK_ISSUER and OIDC_RP_CLIENT_ID` so it stays off until set —
  BN keeps its current local login untouched otherwise.
- Pairwise `sub` needs **no app change**: the Keycloak-side mapper means allauth receives an
  already-pairwise `sub` as `SocialAccount.uid`.
- **Audience note**: allauth doesn't hard-validate `aud` the way mozilla-django-oidc does; the
  token is JWKS-validated. If strict `aud`/`azp` enforcement is required (ADR), add a check.

## Env (BN's DigitalOcean app)

```
KEYCLOAK_ISSUER=https://id.beauaccesssolutions.com/realms/bas
OIDC_RP_CLIENT_ID=benefits-navigator-web
OIDC_RP_CLIENT_SECRET=<BN_CLIENT_SECRET from secrets.env / the realm>
```

## Logistics

- **Worktree, not the shared checkout.** The BN checkout is flagged as possibly modified
  concurrently — do this work in a git worktree off `origin/main`.
- **Data (optional):** export existing verified-email BN users → Keycloak partial-import to
  pre-provision accounts (mirrors KindredAccess's `export_keycloak_users`), per decision #2.

## Verification (exit criterion)

`https://vabenefitsnavigator.org/accounts/oidc/keycloak/login/` 302s to
`https://id.beauaccesssolutions.com/realms/bas/.../auth?client_id=benefits-navigator-web…`
(PKCE), Keycloak renders the login page, a round-trip lands back authenticated, and a
same-verified-email SSO user links to (not duplicates) the existing local account.
